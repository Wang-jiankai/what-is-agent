# 03 | 记忆（Memory）

## 🎯 学习目标

- 理解 Agent 的两类记忆机制：会话上下文 vs 持久化记忆
- 掌握 Claude Agent SDK 的消息历史管理方式
- 了解向量数据库 + RAG 的基本原理
- 能区分"当下记忆"和"历史记忆"的使用场景

---

## 📖 概念讲解

### 什么是 Agent 的记忆？

人类大脑能记住两种东西：

| 类型 | 例子 | 特点 |
|------|------|------|
| **工作记忆** | 现在正在想的事情 | 容量有限，随时间消退 |
| **长期记忆** | 童年的事、昨天发生的事 | 容量大，能保存很久 |

Agent 同样有两种记忆机制：

| 类型 | 实现方式 | 特点 |
|------|---------|------|
| **会话上下文**（工作记忆） | LLM 的 Context Window | 容量有限（取决于模型），本会话有效 |
| **持久化记忆**（长期记忆） | 向量数据库 + RAG | 容量无限，跨会话有效 |

---

## 上篇：会话上下文（Context Window）

### 什么是 Context Window？

Context Window（上下文窗口）是 LLM 一次性能够处理的"工作记忆"空间。你可以把它理解为一块**黑板**——LLM 一次只能在黑板上写下有限的内容。

Claude 3.5 的上下文窗口是 **200K tokens**，大约相当于：
- 15 万个汉字
- 或一部《战争与和平》的文字量

### 消息历史的组织方式

Claude Agent SDK 以**消息列表**的形式管理对话历史：

```typescript
// 消息历史示例
const messages = [
  {
    role: "user",
    content: "我叫王小明"
  },
  {
    role: "assistant",
    content: "好的，王小明，很高兴认识你！"
  },
  {
    role: "user",
    content: "我昨天做了什么？"
  }
];
```

每调用一次 `query()`，SDK 会把历史消息一起发送给 LLM，让它"记得"之前说过什么。

### 会话上下文的局限性

**容量有限**：当对话变长，消息总量超过 Context Window 时：

```
┌─────────────────────────────────┐
│  Context Window（200K tokens）  │
├──────────────┬──────────────────┤
│  早期消息     │   最新消息        │
│  （可能被    │   （保留）        │
│   截断）      │                  │
└──────────────┴──────────────────┘
```

早期的消息会被"遗忘"。这是一种**主动遗忘**机制——Agent 总是保留最相关的最近消息。

### Token 限制的影响

| 对话长度 | 会发生什么 |
|---------|-----------|
| 短对话 | 所有历史完整保留，Agent 完全记得 |
| 中等长度 | 早期消息被截断，Agent 可能忘记用户名字等早期信息 |
| 超长对话 | 只保留最近几轮，需要外部记忆补充 |

## 下篇：持久化记忆（RAG + 向量数据库）

### 为什么需要持久化记忆？

会话上下文的问题：
- **容量有限**：超出就遗忘
- **会话隔断**：关闭对话就消失
- **跨会话丢失**：今天记住的，明天就没了

现实场景需要：Agent 能记住"三个月前用户说过他喜欢什么"，或者"去年项目遇到的技术难点是什么"。

### 向量数据库是什么？

普通数据库按**精确匹配**查询："找 title = '张三' 的记录"

向量数据库按**语义相似度**查询："找和'咖啡'意思最相近的 5 句话"

```
文本 → Embedding（向量化）→ 数学向量
                                    ↓
用户查询 → 向量化 → 在向量空间中找最近邻
```

向量数据库就是专门存储和检索这种"数学向量"的数据库。

### RAG 流程

**RAG = Retrieval Augmented Generation（检索增强生成）**

```
用户问题
    │
    ▼
┌─────────────┐    检索最相关的几条记忆
│  向量数据库   │ ──────────────────────►
│  (Chroma/   │                        │
│   FAISS)    │◄────────────────────── │
└─────────────┘    返回相关记忆内容     │
                                     │
                                     ▼
                              拼接成 prompt
                                     │
                                     ▼
                              LLM 生成回答
```

### 主流向量数据库对比

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| **ChromaDB** | 轻量、易用、有内存模式 | 快速原型、教学 |
| **FAISS** | Facebook 出品，性能强 | 大规模生产环境 |
| **Pinecone** | 云服务、完全托管 | 企业级生产 |
| **Milvus** | 开源、云原生 | 超大规模数据 |

## 串联：两种记忆的配合

真实系统通常**两者都要**：

```
┌─────────────────────────────────────────────┐
│              Agent 的完整记忆系统              │
├──────────────────┬──────────────────────────┤
│  会话上下文        │  持久化记忆               │
│  (Context Window) │  (Vector DB + RAG)      │
├──────────────────┼──────────────────────────┤
│ · 当前任务上下文   │ · 用户偏好历史            │
│ · 本次对话历史     │ · 领域知识               │
│ · 最近几步推理     │ · 跨会话经验              │
└──────────────────┴──────────────────────────┘
         ↑                        ↑
         │ 注入                   │ 检索
         └──────────┬────────────┘
                    ▼
              SystemPrompt
                    │
                    ▼
               LLM 的大脑
```

| 问题 | 解决方式 |
|------|---------|
| "小明刚才说他叫什么名字？" | 会话上下文（直接记得） |
| "用户三个月前说喜欢什么咖啡？" | 持久化记忆（RAG 检索） |
| "当前任务的背景是什么？" | 会话上下文 |
| "这个项目以前遇到过什么问题？" | 持久化记忆 |

## 💡 生活中的类比

**厨房的两种记忆：**

| 机制 | 人类对应 | 说明 |
|------|---------|------|
| 会话上下文 | 主厨在砧板上的工作区 | 手边的东西随时能用，但只能放这么多 |
| 持久化记忆 | 主厨的笔记本 | 记在纸上，下次翻开来还能用 |

主厨工作时：一边看砧板上的当前食材（会话上下文），一边翻笔记本查昨天的订单（持久化记忆）。

---

## 📝 章节回顾

**记住这三个关键点：**

1. **Agent 有两种记忆**
   - 会话上下文：Context Window 内的消息历史，本会话有效
   - 持久化记忆：向量数据库 + RAG，跨会话有效

2. **会话上下文会截断**
   当消息过长时，早期内容被遗忘，越新的消息越可能被保留

3. **RAG 是长期记忆的标准做法**
   Embedding → 存储 → 检索 → 注入 Context → LLM 回答
